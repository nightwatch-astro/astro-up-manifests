# Implementation Plan: Manifest Pipeline Modernization

**Branch**: `001-manifest-pipeline` | **Date**: 2026-03-29 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-manifest-pipeline/spec.md`

## Summary

Build a TOML manifest format, Rust compiler (TOML → SQLite), and version checker with 9 check methods for astrophotography software packages. The compiler produces a signed `catalog.db` with 8 normalized tables and FTS5 search. The checker discovers new versions in parallel with rate limiting, exponential backoff, and persistent failure tracking. A single CI pipeline runs on a 6-hour cron schedule.

## Technical Context

**Language/Version**: Rust (stable toolchain, 2021 edition)
**Primary Dependencies**: serde, toml, rusqlite (bundled), reqwest, tokio, chromiumoxide, clap
**Storage**: SQLite (catalog.db), JSON files (versions/), JSON state file (checker-state.json)
**Testing**: `cargo test` — integration tests with real TOML files and SQLite databases
**Target Platform**: Linux (CI runner), macOS/Linux (local development)
**Project Type**: Two CLI binaries in a Cargo workspace
**Performance Goals**: Compile 95+ manifests in <5s, check all manifests in <15min
**Constraints**: CI runner standard resources, 6-hour cron interval, Chromium binary for browser_scrape
**Scale/Scope**: 95 manifests initially, designed for 200+

```text

versions/                      # Per-version JSON files (versions/{id}/{ver}.json)
checker-state.json             # Persistent failure tracking state

crates/
├── compiler/
│   ├── Cargo.toml
│   └── src/
│       ├── main.rs            # CLI entry point (clap)
│       ├── manifest.rs        # TOML manifest types + serde
│       ├── validate.rs        # Semantic validation (post-deserialization)
│       ├── schema.rs          # SQLite schema creation + FTS5
│       ├── compile.rs         # TOML → SQLite compilation logic
│       └── version_file.rs    # Version JSON file reading + aggregation
│
├── checker/
│   ├── Cargo.toml
│   └── src/
│       ├── main.rs            # CLI entry point (clap + tokio)
│       ├── providers/
│       │   ├── mod.rs         # CheckProvider trait + dispatch
│       │   ├── github.rs      # GitHub Releases API
│       │   ├── gitlab.rs      # GitLab Tags API
│       │   ├── direct_url.rs  # Fetch URL + regex extraction
│       │   ├── http_head.rs   # HEAD request + header extraction
│       │   ├── html_scrape.rs # reqwest + scraper
│       │   ├── browser_scrape.rs  # chromiumoxide
│       │   ├── pe_download.rs # Download + goblin PE parsing
│       │   └── manual.rs      # Skip + log
│       ├── version.rs         # Version parsing (semver, date, custom)
│       ├── hash.rs            # Hash discovery (url+regex, jsonpath, download)
│       ├── state.rs           # checker-state.json read/write
│       ├── rate_limit.rs      # Per-provider rate limit tracking
│       └── issue.rs           # GitHub issue auto-create/close
│
└── shared/
    ├── Cargo.toml
    └── src/
        ├── lib.rs
        ├── manifest.rs        # Shared manifest types (re-exported)
        └── template.rs        # $version variable substitution

.github/workflows/
└── pipeline.yml               # Single-job CI: check → compile → sign → publish
```

**Structure Decision**: Three crates in the workspace — `compiler` (sync, SQLite-focused), `checker` (async, network-focused), and `shared` (manifest types and template engine used by both). This avoids duplicating manifest types while keeping heavy dependencies (chromiumoxide, reqwest) isolated in the checker.
